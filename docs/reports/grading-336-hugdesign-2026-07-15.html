Answer key ไม่มีข้อมูลให้เทียบ แต่ pipeline จัดการความกำกวมได้ดี

โฟลเดอร์ เงินกู้/ ใน answer key ว่างเปล่าสนิท (ไม่มีไฟล์) ค้นยอดที่เกี่ยวกับเงินกู้ทั้งหมด (10,000 / 30,000 / 50,000 / 100,000 / 150,000 / 230,000 / 11,619.50) ในทุกไฟล์ answer key แล้ว ไม่เจอเลย — ร่องรอยเดียวคือรายการเงินสด/ธนาคารทั่วไปใน Statement ซึ่งตรง 100% อยู่แล้วในหัวข้อ ledger ด้านบน

pipeline พบและธง 8 รายการที่เกี่ยวกับเงินกู้เอง แบ่งเป็น 2 ทิศทางที่ pipeline สังเกตเห็นความขัดแย้งด้วยตัวเอง: 4 สลิปโอนเงิน (เบิก OD เข้า 150,000/50,000, จ่ายคืน 100,000, จ่ายดอกเบี้ย+ประกัน 11,619.50) กับ 4 หน้าสัญญากู้ยืมที่บันทึกทิศทางตรงข้าม (บริษัทเป็นผู้ให้กู้ ไม่ใช่ผู้กู้) — ทุกรายการถูกตั้ง confidence: low/medium, needs_review: true และ pipeline เปิดคำถามชัดเจนถึงความขัดแย้งทิศทางผู้กู้/ผู้ให้กู้ระหว่างเอกสาร 2 ชุดที่วันที่ตรงกัน รวมถึงจับได้ว่าตัวเลขไทยเขียนกำกับ (230,000 ซ้ำทั้ง 4 หน้า) ไม่ตรงกับตัวเลขอารบิกที่พิมพ์ (50,000/10,000/230,000/30,000) — เป็นข้อบกพร่องของเอกสารต้นฉบับจริง ไม่ใช่ pipeline อ่านผิด

ความเสี่ยงที่ควรระวังก่อน export จริง: 4 กลุ่มสัญญากู้ยืมยังเป็น placeholder ID_NOT_FOUND ที่ยังไม่ยืนยัน category — ถ้า export ออกไปก่อนคนยืนยัน อาจนับซ้ำกับรายการ ledger ที่จับคู่ไปแล้ว ธงที่ pipeline ตั้งไว้เองคือเกราะป้องกันตรงนี้ ต้องเคารพก่อน export

มี 9 ไฟล์เท่านั้นที่ไม่ถูกอ้างอิงในกลุ่มเอกสารหรือ segment ใดเลย — ทั้ง 9 เป็นไฟล์รายงาน/listing จากแพลตฟอร์ม (shopee.xlsx, tiktok.xlsx, SimpleSalesReport, KPI overview.xlsx, รายงาน Lazada.pdf, ภาษีซื้อ.xlsx, WithholdingTaxReport (1).xlsx, สรุปยิงแอด facebook.xlsx, รายงานชำระ COD) ซึ่ง Stage 1 เขียนไว้ใน _segments/SUMMARY.md เองอยู่แล้วว่าเป็น "รายงานสรุป ไม่ใช่แหล่งคีย์รายการ ไม่ควรแตกไฟล์อ่านทีละบรรทัด" — 1 ในนั้น (สรุปยิงแอด facebook.xlsx) มีธงรอคนยืนยันอยู่แล้วด้วย ส่วนนี้ทำงานตามที่ออกแบบไว้ถูกต้อง

สรุปรูปแบบที่ควรแก้ — เรียงตามความสำคัญ

  1. บั๊กแปลงสกุลเงินรายได้ต่างประเทศ — ยอด THB ที่แปลงถูกแล้วมีอยู่ใน description แต่ไม่ถูกเขียนลง facts.gross_total เป็นบั๊กเชิงกลไก แก้ตรงจุดได้ กระทบ 16% ของตัวอย่างรายได้ต่างประเทศ
  2. เงินกู้ OD หลุดเข้า doc-group tree ฝั่งรายได้ — 3 กลุ่มยืนยันแล้ว ฿290,000 ควรแยกออกจากรายได้ตั้งแต่ชั้นจัดกลุ่ม
  3. ใบวางบิลรวมยอดเสี่ยงคีย์ซ้ำ — ~9 กลุ่ม ฿197,517 ต้องเช็คว่าใบกำกับภาษีย่อยถูกคีย์แยกไปแล้วหรือยัง ก่อน export ใบวางบิล
  4. Doc-group category ไม่ตามบัญชีที่ confirm แล้ว — กลุ่มเงินกู้ที่ poirot mapping ไปบัญชีงบดุลถูกต้องแล้ว แต่ folder ยังอยู่ที่ expense/non_vat ทำให้ scope ของ bucket นี้บวมผิดปกติ ควรมีขั้นตอน sync path ให้ตรงบัญชีที่ยืนยันแล้ว
  5. งานจ้างผลิต/ติดตั้งสับสนกับ COGS — บัญชี 510xxx (ต้นทุนการผลิต/ซื้อสินค้า) ถูกใช้แทน 530xxx (ค่าบริการ SG&A) ซ้ำหลายรอบ ควรไล่ตรวจกว้างขึ้นทั้ง VAT/non-VAT bucket
  6. วันที่เอกสารแพลตฟอร์ม (TikTok/Shopee) คลาดเคลื่อนราวหนึ่งเดือน — ต้องตรวจว่าอ่านวันที่บันทึกหรือวันที่ตัดรอบ settlement
  7. จุดที่ดี: ทุกที่ที่จับคู่ได้จริง (รายได้ในประเทศ, ledger ธนาคาร, PND3 ส่วนใหญ่) รหัสบัญชีถูก 100% — จุดอ่อนของรอบนี้อยู่ที่การจัดการวันที่/สกุลเงิน/scope ไม่ใช่การจัดหมวดบัญชีเอง
จัดทำหลังรัน pipeline แบบ blind ต่อ answer key ครบตามเกต Ledger Gate แล้วเท่านั้น ตามกติกาใน CLAUDE.md — ไม่มีการดู answer key ระหว่างรัน
ตรวจโดย 2 agent คู่ขนาน (รายได้+ค่าใช้จ่าย, บัญชีเดินสะพัด+เงินกู้) ด้วยสคริปต์ดึงข้อมูล+จับคู่ bulk แล้วผู้ตรวจหลักตรวจทานตัวเลขซ้ำอีกชั้นก่อนสรุปในเอกสารนี้ — พบและแก้ 1 จุดที่ sub-agent อ่านคอลัมน์ผิดระหว่างตรวจ (ดูหมายเหตุด้านบน)
