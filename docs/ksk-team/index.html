The ksk-keying Agent Team

Six bounded, memoryless leaf subagents and one orchestrating parent turn a raw Thai client document folder into reviewable, PEAK-importable bookkeeping — with two human gates and zero guessing on ambiguous evidence. This page explains who does what, in what order, and the two invariants that are easiest to break when you touch this system.

1. Executive Summary

Parent orchestrates

One parent session holds all workflow state and dispatches every stage as an Agent call. It never reads, interprets, links, or maps a document itself.

Leaf-only children

Every subagent is bounded to one unit of work (one segment, one group, one client) and cannot spawn further subagents.

Two hard invariants

Bookable-doc-vs-evidence linking, and the parent's zero-document-work rule. Both are subtle, both corrupt the books or blow the context budget if broken.

Human stays in the loop

AI output is always a proposal. Three explicit gates — client profile, segmentation, transaction linking — plus mandatory final review before PEAK export.

What this system actually produces

A per-bucket review.html the human opens in a browser — one interactive page per expense/vat, expense/non_vat, income/vat, bank_statement, etc. — previewing the real source document next to the AI's proposed account mapping, with a one-click peak_import_<bucket>.xlsx export for PEAK accounting software. Nothing downstream of the AI stages is treated as final bookkeeping truth until a human has looked at it.

4. Agent Detail Cards

🕵️ ksk-magnumStage 0 · First contact
model: sonnet tools: Read, Glob, Grep, Bash, Write writes: CLIENT.md, coa.csv

Role. The first-contact investigator. Builds the client profile every downstream stage leans on — who this client is, and how their books tend to work — then hands the parent a short list of things only a human can confirm. Never processes documents for accounting facts.

Job, in order

  • Ensure context files exist. coa.csv is required — poirot's only valid source of account codes. If missing, convert it from the client's ผังบัญชี chart-of-accounts workbook with the deterministic tool bun run --cwd .claude/skills/ksk-keying/scripts coa-to-csv — never hand-transcribed. If no workbook exists either, that's a blocking gate, not a guess.
  • Identify the client from the folder name (usually _<id> <thai company name>) and, if needed, the buyer block of one or two invoices — never the whole document.
  • Infer business nature from coa.csv account style and purchase patterns (e.g. a ต้นทุนการก่อสร้าง line signals a contractor). This single field is what lets ksk-poirot map ambiguous purchases consistently instead of thrashing between buckets.
  • Draft COA conventions — line-pattern → account-code defaults, drawn only from codes that exist in coa.csv, each tagged with its assumption so the human can veto it.
  • List every unknown as a concrete question in needs_confirmation — tax id, low-confidence business nature, COGS-vs-repair calls, capitalization threshold, missing COA workbook.

Hard constraints

  • Every account_code named must exist in the client's coa.csv — never fabricate a code or a tax id.
  • Cannot talk to the user directly — needs_confirmation is the only channel; the parent owns the human gate.
  • Does not segment (ksk-columbo), interpret line items (ksk-watson/ksk-marple), link transactions (ksk-sherlock), or map full COA (ksk-poirot).
  • Writes only CLIENT.md and, when converting, coa.csv. Touches nothing else.

Worked example — needs_confirmation

needs_confirmation:
  - "Client tax id (13-digit) — not printed on sampled buyer blocks"
  - "Are hardware/PVC/paint purchases COGS raw material (510111) or facility repair (530211)?"
  - "Fixed-asset capitalization threshold (e.g. printer 6,449)"
🔍 ksk-columboStage 1 · Segment
model: haiku tools: Read, Glob, Grep, Bash, Write writes: _segments/manifest.yaml, SUMMARY.md

Role. Turns one raw client folder into a segment proposal — the structural first cut over a pile of PDFs, transaction folders, and spreadsheets. Runs on haiku because it is pure structural triage, not accounting judgment.

Job, in order

  • Detect the folder's shape: concatenated PDFs, transaction folders, flat 1:1 files, spreadsheets, or a mix.
  • Propose segment boundaries — stable segment_id, source files/page ranges, a type (pdf_range, transaction_folder, single_file, spreadsheet) — with confidence and open questions.
  • Record structural co-location (shared folder, adjacent pages) as a hint only — never assert it means "same transaction"; that call belongs to ksk-sherlock.
  • Flag concatenated multi-document scans. A 75-page PDF holding ~45 separate supplier invoices gets multi_document: true plus an estimated sub-document count, so the parent fans out one ksk-watson read per sub-document instead of one pass over the whole scan.

Hard constraints

  • Never interprets document contents — no reading images/spreadsheets for facts, boundaries only.
  • Never decides same-transaction grouping (ksk-sherlock's job) or COA mapping.
  • Says so explicitly when boundaries are genuinely ambiguous, instead of picking one arbitrarily.

Why this matters

Missing the multi-document flag is the single highest-leverage failure mode in this stage: a child that reads a 75-page scan in one pass loses line-item detail across dozens of invoices, and that loss silently propagates all the way to COA mapping.

👁️ ksk-watsonStage 2 · Interpret (visual)
model: sonnet tools: Read, Glob, Grep, Bash native vision — Read on an image returns the image

Role. Reads exactly one approved visual segment — or one sub-document page range inside a concatenated scan — and returns normalized accounting evidence. Never guesses content or skips a page.

Required workflow

  • Confirm the segment id and image list the parent gave it; check the files exist.
  • 1Classify each document's doc_kind, then 2apply the matching playbook from references/extract-playbooks.md — see §6. The parent never picks a doc-type; Watson classifies as it reads.
  • Interprets document roles, amounts, dates, parties, VAT/WHT, and how images within the segment relate (same transaction vs. separate).
  • Reads CLIENT.md's default_buyer only to reliably tell the client-buyer party from the supplier-seller party — never lets it override what the document actually shows.
  • Records source_file + source_page for every document, so the review UI can later preview the exact page.

Hard constraints

  • Never inspects the whole client — only the unit the parent pointed it at.
  • Never guesses missing facts; surfaces uncertainty in review_flags / questions_for_user instead.
  • Never performs COA mapping. Never writes files unless explicitly asked for one.

Worked example — output shape

{
  "segment_id": "segment-001",
  "documents": [{ "doc_kind": "normal_bill_or_invoice",
    "document_role": "supplier_invoice", "source_page": 5, "confidence": "high" }],

🧩 ksk-sherlockStage 3 · Link
model: sonnet tools: Read, Glob, Grep, Bash, Write writes: _doc_groups/links.yaml

Role. The sole owner of the same-transaction decision. Segmentation groups files only by structure; Watson only sees relationships within one segment. Sherlock is the cross-segment step: which approved segments belong to the same accounting transaction, decided by reading their already-normalized interpretations — never raw images or spreadsheets.

Linking rules — strong evidence only

  • An exact shared document / reference / PO / invoice number, or
  • A matching (amount + date + counterparty) triple, or
  • Columbo's structural co-location — treated as strong evidence, but sanity-checked against the interpreted facts and split back out if contents clearly disagree.

When evidence is weak, partial, or only circumstantial — same seller but different amounts, near-but-not-equal dates — segments stay unlinked and a questions_for_user entry is raised. Never link on a guess: a wrong merge or split corrupts the booking downstream. A segment with no strong match to anything is its own single-member transaction — that's a normal, correct outcome.

Hard constraints

  • Never re-interprets raw documents — consumes interpretations only.
  • Never does COA mapping, doc-group tree building, or review generation.
  • Never concatenates document numbers, and never collapses a multi-invoice payment into one bookable record. See §5 for the full invariant.

Full worked example of a two-invoice-one-receipt link is in §5, since it's the canonical illustration of the bookable-doc invariant.

🎩 ksk-poirotStage 5a · Categorize
model: sonnet tools: Read, Glob, Grep, Bash, Write writes: categorize.json

Role. Proposes chart-of-accounts mappings for one doc group's line items. Poirot proposes; a human confirms. Wrong or over-confident mappings corrupt the books, so it stays conservative by design.

Matching priority, strongest first

  1. Exact tax_id match against coa_usage.json hints — strongest signal.
  2. A confirmed CLIENT.md convention or a coa_usage.json keyword hint for the line's category.
  3. Description keywords on the line item.
  4. Account-name semantics as the fallback.

A CLIENT.md convention is still a hint, not proof: the chosen code must exist in coa.csv, and a document's own tax_id evidence beats a general convention when they conflict.

Hard constraints

  • Every code must exist in the client's coa.csv — never fabricate a code or account name.
  • One doc group per call; never reads other groups or re-interprets raw documents.
  • Conservative on ambiguity: sets needs_review: true rather than guessing confidently.

Worked example — one categorize.json line

{ "line_index": 2, "account_code": "510111", "sub_code": null,
  "account_name_th": "ซื้อวัตถุดิบ", "confidence": "medium",
  "reason": "Matches CLIENT.md convention: PVC/paint/hardware treated as production input",
  "needs_review": false }
🕊️ ksk-marpleStages 2, 4a, 4b, 5b · Multi-purpose worker
model: sonnet tools: Read, Write, Edit, Glob, Grep, Bash writes: interpretation.json, manifest.yaml, review-data.json

Role. The bounded worker for every ksk-keying step that isn't folder scouting, visual reading, linking, or COA categorize. The parent tells it exactly which of four tasks to do — it does only that one, never drifting into the next stage.

The four tasks, one per call

  • Spreadsheet/report interpretation — parses xls/xlsx/csv for one approved segment into the same interpretation shape visual segments use.
  • Doc-group skeleton (4a) — builds the _doc_groups/ tree + manifest.yaml only, structural, no line-item transcription. One links.yaml cluster with N bookable_docs becomes N group folders, never one merged folder.
  • Doc-group populate (4b) — for one group folder, writes interpretation.json by copying the full facts and every line item from the upstream segment interpretation. Never collapses a purchase bill to just vendor + invoice number.
  • Review-data build (5b) — for one group folder, writes review-data.json from interpretation.json + categorize.json. Sets source_src/source_page so the review UI opens the real source document at the right page; backfills buyer/buyer_tax_id from CLIENT.md's default_buyer when the document itself doesn't print it. Never runs the HTML generator itself — that's the parent's one deterministic shell command.

Hard constraints

  • One segment, one group, or one skeleton pass per call — never the whole client.
  • Never guesses missing facts — sets needs_review: true, initial_status: needs_attention instead.

5. The Two Invariants

Everything else in this system is a scoping rule. These two are correctness rules — get either wrong and the output looks fine until a human (or the Thai Revenue Department) finds the discrepancy.

Invariant A — Bookable document vs. evidence (owned by ksk-sherlock)

The bookable unit is the primary document number. Evidence links documents; it never merges document numbers. One ใบกำกับภาษี (tax invoice) = one bookable document = one booking, always. This is atomic and is never subdivided or fused.

Two orthogonal axes, and the failure mode is conflating them:

  • Bookable unit — the primary document number. Never subdivided or fused, no matter how the documents are physically bundled.
  • Evidence — shared reference numbers, matching amount sums, receipts, bank withdrawals, WHT certificates. Evidence expresses how documents relate; it never merges their document numbers into one booking.

A transaction cluster is therefore a set of related documents that declares a list of bookable documents — one entry per primary document number — plus the evidence tying them together. The count of bookable_docs always equals the count of distinct primary document numbers in the cluster.

Worked example — two invoices, one receipt

- transaction_id: txn-008
  segments: [segment-015, segment-017, segment-018]
  members:
    - {segment: segment-015, document_no: IVT-20260300028, role: primary_invoice}  # meter usage
    - {segment: segment-017, document_no: IVT-20260300029, role: primary_invoice}  # copier lease
    - {segment: segment-018, document_no: RE-20260400007,  role: payment_receipt}
  bookable_docs: [IVT-20260300028, IVT-20260300029]   # TWO bookable units, NOT "028 + 029"
  evidence: "Both invoices (201.59 meter + 2,675.00 lease) sum to 2,876.59 = receipt
             RE-20260400007 = bank withdrawal 27-04-26. Different WHT rates (3% vs 5%)
             mean each invoice must be booked and withheld separately."
  confidence: high

Why it's non-negotiable. Thai input VAT (ภาษีซื้อ / ภ.พ.30) is reported per tax-invoice number, and separate invoices can carry different VAT bases and WHT rates. Fusing two document numbers into one record corrupts both the VAT report and the WHT certificate — this is exactly the kind of error that looks fine in a spreadsheet and only surfaces at filing time. It's why ksk-sherlock hard-bans string concatenation like "INV-A + INV-B" and a single collapsed bookable record, and why Stage 4a creates one group folder per bookable_docs entry rather than per transaction cluster.

Invariant B — The parent never does the work

The parent session does zero document work. Every stage runs inside a subagent via the Agent tool. The parent only dispatches children, holds state between stages, runs the single deterministic shell command, and stops at human gates.

What the parent does

Decides which stage runs next and dispatches the right Agent call(s)
Holds cross-stage state (segment ids, group paths, client path)
Fans out every independent unit of a stage in one message, in parallel
Runs bun run --cwd .claude/skills/ksk-keying/scripts review-groups once, directly (the only non-agent action)
Stops and asks the human at each 🚦 gate

What the parent never does

Read a document image or spreadsheet for accounting facts
Decide same-transaction grouping or COA codes itself
Write interpretation.json, categorize.json, or review-data.json itself
Let a child spawn its own subagents
Restate an agent's own rules in the dispatch prompt

Two mechanical consequences follow directly from this rule:

  • Dispatch prompts are caveman-short. Each agent's own .md file already holds the full how-to; the prompt carries only the variable data — task tag, client path, exact ids, exact file paths. Nothing else, because every extra sentence in the prompt is a sentence the parent is duplicating from the agent definition it should trust.
  • Children have no memory. Every dispatch must be self-contained, because a child agent cannot recall anything from a prior call or from the parent's own context.

The reason this rule exists is context budget: the parent orchestrating a whole client's worth of documents would blow its own context window if it read even a fraction of what its children read. Delegation isn't a style preference here — it's the thing that makes the workflow able to run at all on large clients.

6. Watson's Extraction Playbooks

ksk-watson reads every Thai accounting document with the same generic instinct unless it's told otherwise — which loses the hard rules each document type needs (a PEA electricity bill's real total sits in a lower block; a WHT certificate's "seller" is actually the withholder, not the preparer). references/extract-playbooks.md encodes those rules, distilled from eval-validated extraction prompts. Watson classifies doc_kind first, then applies the matching playbook — the parent never picks the doc-type for it.

doc_kindSignalNotable quirk
handwritten_billDetails mainly handwritten on a cash bill/receipt/preprinted tax-invoice formExtract only the printed item grid; ignore free-write margin notes. Use "ไม่สามารถระบุได้" rather than guess a messy word — never infer a product name from shape/qty/seller type.
pwa_billการประปา / Provincial Waterworks AuthoritySpecialized total/field layout distinct from a generic invoice.
pea_billการไฟฟ้าส่วนภูมิภาค / Provincial Electricity AuthorityThe real total lives in a lower block on the form — a generic reader grabs the wrong number.
global_house_invoiceGlobal House receipt / tax invoice / purchase documentRetailer-specific layout with its own field mapping.
wht_certificateหนังสือรับรองการหักภาษี ณ ที่จ่ายThe document's "seller" is the party who withheld tax, not the preparer — a generic read gets the party roles backwards.
delivery_noteใบส่งของ anywhere in the titleHard override: classified as delivery_note even when it also carries tax-invoice markings and a single line.
bank_statementBank statement / account movement report / passbook pageFeeds the chronological bank_statement review schema, not the invoice-shaped one.
normal_bill_or_invoiceOrdinary printed receipt/bill/invoice/tax invoice fitting no specialized kindThe default path once every specialized kind is ruled out.
genericUsable accounting page fitting no specialized kind (quotation, PO, credit/debit note, payment page)Fallback playbook for pages that are real evidence but not a named kind above.

Priority rule: specific beats generic — handwritten_bill, pwa_bill, pea_bill, global_house_invoice, wht_certificate, and delivery_note are chosen before falling back to normal_bill_or_invoice. A page that isn't an accounting document at all (cover sheet, index, blank) is flagged as not a usable segment rather than forced into a kind.

AI outputs are proposals, not final bookkeeping truth. Every file above review.html in this tree is an AI proposal. Human review through review.html remains mandatory before any PEAK export, regardless of how high the AI's confidence scores are.

8. Human Gates & Stop Rules

GateTriggerParent action
🚦 After Stage 0ksk-magnum's needs_confirmation list is non-emptyAsk the human (tax id, business nature, COGS-vs-repair calls, capitalization threshold, missing COA workbook), then patch CLIENT.md before Stage 1. Skip only when the list is empty or already answered in an existing CLIENT.md.
🚦 After Stage 1Segmentation confidence is low, groupings are plausible more than one way, files are missing, or the child raises questions that change downstream meaningStop for the user. Otherwise continue.
🚦 After Stage 3A link is ambiguous or would merge/split on weak evidenceStop for the user. Skip this stage's gate only when every transaction lives fully inside one segment.
🚦 Final, alwaysEvery run reaches this point regardless of confidenceHuman opens each _doc_groups/<bucket>/review.html via file://, reviews line-by-line against the previewed source document, and exports peak_import_<bucket>.xlsx.

General stop rule

Escalate to the user instead of guessing whenever: segmentation is ambiguous, a required source file or page image is missing, a child needs a product decision outside its task, an accounting interpretation lacks enough evidence, or a group can't be mapped conservatively. Park unresolved output where a human can review it — never silently finalize a low-confidence result.

The agent definition .md files are each subagent's system prompt when dispatched via subagent_type. Editing one changes that agent's behavior everywhere it's invoked — there is no per-call override except the dispatch prompt's variable data.

Dispatch prompt templates

Every stage's dispatch prompt is short by design — the agent's own .md already has the how-to, so the prompt carries only variable data. These are the exact templates the orchestrating skill specifies:

// Stage 0 — client profile
Agent({ description: "Client profile", subagent_type: "ksk-magnum",
  prompt: `First-contact profile for client "${clientPath}". Write CLIENT.md.` })

// Stage 1 — segment
Agent({ description: "Segment", subagent_type: "ksk-columbo",
  prompt: `Segment client "${clientPath}". Write _segments/manifest.yaml + SUMMARY.md.` })

// Stage 2 — interpret, visual segment
Agent({ description: "Read visual", subagent_type: "ksk-watson",
  prompt: `Segment ${segmentId}. Client "${clientPath}". Images: ${imagePaths}. Related: ${relatedFiles}.` })

// Stage 2 — interpret, multi-document scan sub-document (fan out one per page range)
Agent({ description: "Read invoice", subagent_type: "ksk-watson",
  prompt: `Sub-document of ${segmentId}. Client "${clientPath}". Source: ${pdfPath} pages ${pageRange}.
           Read only these pages; report source_file + source_page.` })

// Stage 2 — interpret, spreadsheet segment
Agent({ description: "Read sheet", subagent_type: "ksk-marple",
  prompt: `spreadsheet interpretation. Segment ${segmentId}. Client "${clientPath}". Files: ${filePaths}.` })

// Stage 3 — link
Agent({ description: "Link", subagent_type: "ksk-sherlock",
  prompt: `Link segments for client "${clientPath}". Interpretations: ${segmentResultsSummary}.
           Write _doc_groups/links.yaml.` })

// Stage 4a — group skeleton
Agent({ description: "Group skeleton", subagent_type: "ksk-marple",
  prompt: `doc-group skeleton. Client "${clientPath}". Links: _doc_groups/links.yaml.
           Interpretations: ${segmentResultsSummary}. Write _doc_groups/manifest.yaml + the
           category/VAT tree + empty group folders. Create ONE group folder per bookable_docs
           entry, not per transaction. Do not populate interpretation.json.` })

// Stage 4b — group populate (fan out one per group)
Agent({ description: "Group populate", subagent_type: "ksk-marple",
  prompt: `doc-group populate. Group "${groupPath}". Client "${clientPath}".
           Source interpretation: ${segmentInterpretationPath}.
           Write ${groupPath}/interpretation.json with full line items + source_ref/source_page.` })

// Stage 5a — categorize (fan out one per group)
Agent({ description: "Categorize", subagent_type: "ksk-poirot",
  prompt: `Categorize group "${groupPath}". Client "${clientPath}". Write categorize.json.` })

// Stage 5b — review-data (fan out one per group, after that group is categorized)
Agent({ description: "Review-data", subagent_type: "ksk-marple",
  prompt: `review-data. Group "${groupPath}". Client "${clientPath}".` })

// Stage 5c — generate HTML (parent runs directly, once, after all review-data.json exist)
bun run --cwd .claude/skills/ksk-keying/scripts review-groups -- --force "${clientPath}"

review-groups.ts hard-errors if a group folder's review-data.json doesn't match the schema expected for its bucket, so a malformed write here fails loudly at generation time rather than silently producing a broken preview.

Edge cases this system explicitly handles

Multi-document scans

A single PDF holding dozens of separate invoices must be flagged by ksk-columbo (multi_document: true) and fanned out one ksk-watson call per sub-document/page-range — never read as one pass, which loses line-item detail.

Mixed VAT documents

A document whose line items mix VAT and non-VAT lines gets the mixed bucket under expense/ (rare/flagged under income/) rather than forcing an all-or-nothing VAT classification.

N-invoices-one-payment

See Invariant A (§5) — must yield N group folders and N bookable_docs entries, with the shared receipt recorded as evidence on all of them, never as its own bookable unit.

Missing chart of accounts

If neither coa.csv nor a ผังบัญชี workbook exists, ksk-magnum records this in needs_confirmation as a blocking condition — the workflow cannot proceed past Stage 0 without one.

Document without a printed buyer tax id

ksk-marple's review-data build backfills facts.buyer/facts.buyer_tax_id from CLIENT.md's default_buyer rather than leaving it null, because the PEAK export needs the buyer id.

Weak or circumstantial linking evidence

ksk-sherlock leaves segments unlinked and raises a questions_for_user entry rather than guessing — a wrong merge/split is treated as worse than an unresolved question.

Acceptance criteria for a correct run

Required artifacts exist for every stage actually run, in the order listed in §7
Every child stayed inside its bounded scope (one segment / one group / one client) and did not spawn subagents
No child owned workflow state — only the parent held cross-stage state
Every bookable_docs entry across all clusters has its own group folder — count matches, never merged
Every non-empty bucket has a generated review.html + assets/ after Stage 5c
Every review-data.json page has a valid source_src/source_page the reviewer can actually open
Human review remains the last control point before any PEAK export — no stage treats its own output as final
