2. ทำไมต้องมี — บทเรียนที่จุดชนวน

ตอนเทียบงานเดือนพฤษภาคมของลูกค้า 216 กับเฉลย เราสรุปตอนแรกว่า "watson ลืมดึง WHT 691.88 ของเอกสาร 033" แต่พอเปิดไฟล์จริงดู:

// interpretation.json ของเอกสาร 033 (ที่ watson เขียนจริง)
"wht": null,
"review_flags": [
  "wht_expected?: rent and service charges from a juristic-person landlord
   (Golden Land Property Development PLC) but the invoice shows no WHT
   column or amount — WHT may be settled separately (PND53)..."
]

เอกสารไม่ได้พิมพ์ WHT จริงๆ — watson ทำถูกตาม contract ("จดเฉพาะที่เห็น ห้ามคำนวณเอง แล้วยกธง") จุดพังคือปลายน้ำไม่มีใครแปลงธงเป็น booking ถ้าวันนั้นเราเชื่อความรู้สึกแล้วไป "แก้" prompt ของ watson เราจะทำลาย rule ที่ถูกอยู่แล้ว เพื่อซ่อนบั๊กที่อยู่คนละชั้น

Evals เกิดมาเพื่อกันเหตุการณ์แบบนี้: ก่อนแตะ prompt ใคร ต้องมีตัวเลขบอกว่าตัวนั้นผิดจริงหรือเปล่า และหลังแก้ ต้องมีตัวเลขบอกว่าดีขึ้นจริงโดยไม่ทำของเดิมพัง

4. วงจรการรัน — harvest → dispatch → grade → baseline

harvest.tsสร้างเคสจาก run ที่ verify แล้วเท่านั้น (ledger gate ต้อง pass)
→
dispatch.tsสร้าง run dir + พิมพ์ prompt ตาม template production เป๊ะ
→
agent จริง ×Nspawn ขนานผ่าน Agent tool — harness เดียวกับ production
→
grade.tsตรวจ deterministic รายฟิลด์ ไม่มี LLM-judge
→
report.tsตาราง + diff กับ baseline, exit 1 เมื่อ regress
cd evals
bun run dispatch.ts -- watson --note "ทดลองอะไรอยู่"   # สร้าง run + พิมพ์ dispatch prompt
# (parent session spawn ksk-watson ตาม prompt ที่พิมพ์ — ขนานทุกเคส)
bun run grade.ts    -- watson --run <run-id>
bun run report.ts   -- watson --run <run-id>              # exit 1 ถ้าแย่กว่า baseline
bun run report.ts   -- watson --run <run-id> --set-baseline

หลักการตายตัวสองข้อ: (1) dispatch prompt มาจาก template เดียวกับ SKILL.md แบบคำต่อคำ — prompt เพี้ยนเมื่อไหร่ตัวเลขไร้ความหมาย (2) model ตัวเดียวใน loop คือ agent ที่ถูกสอบ การตรวจเป็น script ล้วน

2. การเปิดเอกสารจริงดูด้วยตาซูม 600dpi ยืนยัน DATE = 05.05.2026, เลขที่ IVT-20260500096

6. Grading Model — สี่สถานะต่อฟิลด์

สถานะความหมายอันตราย
correctตรงเฉลยใน tolerance (เงิน ±0.01, วันที่/เลขภาษี ตรงตัว, ชื่อบริษัทเทียบแบบ normalize, wht: null≡0)—
wrong_flaggedผิด แต่มี review flag ที่พูดถึงฟิลด์นั้นต่ำ — คนตรวจเห็นธงแล้วไปดู
wrong_silentผิดโดยไม่มีธงสูงสุด — เดินเข้า PEAK โดยไม่มีใครเอะใจ ทั้งสามแบบนี้รวมเป็น silent-error rate
missing / spuriousค่าที่ควรมีแต่หาย / ค่าที่ไม่มีในเอกสารแต่แต่งขึ้น

ฟิลด์ critical ของ watson: document_no · document_date · seller/buyer_tax_id · gross_total · vat · wht · direction · doc_kind · page_disposition (ฟิลด์สุดท้ายคือคำตอบของ "ตัดสิน skip หน้าไหนถูกไหม") ส่วน sherlock วัด cluster_membership · bookable_docs · multi_member_clusters ด้วยการจับคู่ member-set แบบ multiset

ตัวอย่าง report จริง (รัน after-date-rule)

cases: 6/6 pass   solid-only: 3/3 pass   SILENT ERRORS (solid): 0/70 (0.00%)

field              correct  wrong_flagged  wrong_silent  missing  spurious
document_no         37/37          0             0        0         0
document_date       37/37          0             0        0         0
...
wht                 37/37          0             0        0         0
page_disposition    37/38          0             0        1         0   ← จับ harvest gap ได้ทันที

แถว page_disposition ที่ขาด 1 หน้า นำไปสู่การพบว่าเคส b3 ตอน harvest ลืม copy ไฟล์ duplicate มา — บั๊กของคนสร้างข้อสอบ ไม่ใช่คนสอบ แก้เคสแล้ว re-run ได้ 9/9

7. Replicates — วัดความนิ่ง ไม่ใช่แค่ความถูก

--replicates 3 จะ spawn subagent 3 ตัวขนานกัน ด้วย prompt เดียวกัน (context แยกขาด ไม่เห็นคำตอบกัน — ผลทางสถิติเท่ารันตามลำดับ แต่ wall-clock เท่ารอบเดียว) แล้ว grader เพิ่มมิติ agreement รายฟิลด์:

3/3 ตรงกัน

ฟิลด์นิ่ง — เชื่อผลรอบเดียวได้ในอนาคต

2/3

ฟิลด์สั่นคลอน — ปัญหาคือความไม่นิ่งของการอ่าน ไม่ใช่ rule ผิด (วิธีแก้คนละทางกับแก้ prompt)

ต่างกันหมด

อย่าเพิ่งแก้ rule — หา root cause ของความสุ่มก่อน (เช่น ภาพความละเอียดต่ำ)

ค่า token จ่าย ×N ตรงๆ — ใช้เป็น mode เสริมตอนสงสัยความนิ่งหรือตอนจะเคาะ baseline ใหม่ รอบพัฒนา prompt ปกติรันเดี่ยวพอ

8. เคสจริง: ไล่ล่า "วันที่ผี" บนใบเช่าสแกน — 3 รอบจบ

ใบเช่า Golden Land เป็นสแกนความละเอียดต่ำ พิมพ์ DATE 05.05.2026 แต่ watson อ่านเป็น 2026-05-03 เป็นบางรอบ — โดยมั่นใจเต็มร้อย ไม่ยกธง นี่คือ silent error ตัวอย่างตำรา และ eval ทำให้การไล่ล่าเป็นวิทยาศาสตร์:

รอบสมมติฐาน / rule ที่ใส่ผลวัด (3 replicates)บทเรียน
0ยังไม่มี rule2/3 — อ่านเป็น 05-03 หนึ่งรอบจับ failure mode ได้: บางรอบหยิบวันที่ service period ("16.03.2026") บางรอบอ่านเลขผิด
1playbook: document_date = วันที่ออกใน header เท่านั้น ห้ามใช้ช่วงเวลาบริการใน line itemรันเต็ม 6/6 ผ่าน แต่ stability ยัง 2/3ปิดได้หนึ่งประเภท (service period) แต่หลักเลข 05↔03 ยังสั่น
2"ซูมเมื่อรู้สึกไม่แน่ใจ"1/3 — แย่ลง!rule มีเงื่อนไขไม่ทำงาน เพราะการอ่านผิดมันรู้สึกมั่นใจ — agent ไม่เคยรู้สึกว่าต้องซูม
3บังคับซูม: ทุกหน้าสแกน (เช็ค text layer ด้วย pdftotext | wc -c) ต้อง render header 300dpi แล้วอ่านเลขที่+วันที่จากภาพซูมก่อนจด3/3 ทุกฟิลด์ agreeปิดจ๊อบ — r2 ถึงกับรายงานเองว่า "ซูมแล้วยืนยัน 05.05.2026 ไม่ใช่ 03 ที่ความละเอียดต่ำหลอก"

ถ้าไม่มี eval: รอบ 2 ("ซูมเมื่อไม่แน่ใจ") ฟังดูสมเหตุสมผลมาก และคงถูก commit ไปทั้งที่มันแย่กว่าไม่มี rule — ตัวเลข 1/3 เท่านั้นที่หยุดเราไว้

10. Detailed Engineering Spec for Coding Agents

Purpose: ส่วนนี้ละเอียดพอให้ agent ตัวอื่นทำงานต่อได้โดยไม่ต้องสัมภาษณ์ใครใหม่ — โครงไฟล์ สคีมา คำสั่ง และเกณฑ์การเพิ่ม agent ใหม่เข้าระบบ eval

Goal / Non-goals

TypeDescription
Goalวัด agent เดี่ยวบน fixture verified ด้วย grader deterministic; ตัวเลขหลัก = silent-error rate (solid cases); ป้องกัน regression ด้วย baseline diff ที่ exit 1
Goaldispatch prompt ต้อง mirror template ใน SKILL.md แบบ verbatim ผ่าน harness เดียวกับ production (Agent tool)
Non-goalไม่ใช้ LLM-as-judge, ไม่ใช้ promptfoo, ไม่ commit ข้อมูลลูกค้า (dataset/ผลดิบอยู่ใต้ samples/evals/ ซึ่ง gitignored)
Non-goalไม่แทน Ledger Gates หรือ human review ของ production run — eval เป็นเครื่องมือพัฒนา skill

File layout

evals/                          # committed
  lib.ts                        # case loading, interp normalization, flagText, amountEq, normText
  harvest.ts                    # สร้างเคสจาก run ที่ gate=final/pass เท่านั้น (โหมด watson | sherlock)
