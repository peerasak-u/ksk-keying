รายการ 02เหตุจูงใจ — เคสที่เกือบทำให้แก้ผิดตัว

ตอนเทียบงานเดือนพฤษภาคมของลูกค้า 216 กับเฉลย เราสรุปตอนแรกว่า "watson ลืมดึง WHT 691.88 ของเอกสาร 033" แต่พอเปิดไฟล์ที่ watson เขียนจริง:

"wht": null,
"review_flags": [
  "wht_expected?: rent and service charges from a juristic-person landlord
   (Golden Land Property Development PLC) but the invoice shows no WHT
   column or amount — WHT may be settled separately (PND53)..."
]

เอกสารไม่ได้พิมพ์ WHT จริงๆ — watson ทำถูกตาม contract ของมัน ("จดเฉพาะที่เห็น ห้ามคำนวณเอง แล้วยกธง") จุดพังคือปลายน้ำไม่มีใครแปลงธงเป็น booking ถ้าวันนั้นเชื่อความรู้สึกแล้วไป "แก้" prompt ของ watson เราจะทำลาย rule ที่ถูกอยู่แล้ว เพื่อซ่อนบั๊กที่อยู่คนละชั้น

Evals เกิดมาเพื่อกันเหตุการณ์แบบนี้ ก่อนแตะ prompt ใคร ต้องมีตัวเลขบอกว่าตัวนั้นผิดจริงหรือเปล่า — และหลังแก้ ต้องมีตัวเลขบอกว่าดีขึ้นจริงโดยไม่ทำของเดิมพัง

รายการ 04วงจรการรัน — harvest → dispatch → grade → baseline

harvest.tsสร้างเคสจาก run ที่ verify แล้วเท่านั้น (ledger gate ต้อง pass)
→
dispatch.tsสร้าง run dir + พิมพ์ prompt ตาม template production เป๊ะ
→

→

→

# จาก evals/
bun run dispatch.ts -- watson --note "ทดลองอะไรอยู่"
# parent session spawn ksk-watson ตาม prompt ที่พิมพ์ — ขนานทุกเคส
bun run grade.ts    -- watson --run <run-id>
bun run report.ts   -- watson --run <run-id>              # exit 1 ถ้าแย่กว่า baseline
bun run report.ts   -- watson --run <run-id> --set-baseline

หลักตายตัวสองข้อ: (1) dispatch prompt มาจาก template เดียวกับ SKILL.md แบบคำต่อคำ — prompt เพี้ยนเมื่อไหร่ตัวเลขไร้ความหมาย (2) model ตัวเดียวใน loop คือ agent ที่ถูกสอบ การตรวจเป็น script ล้วน

รายการ 05ข้อสอบและเฉลย — expectation ตั้งยังไง

กายวิภาคของหนึ่งเคส

  ├── case.yaml     # โจทย์ + ที่มา + ธงที่ต้องยก

  ├── CLIENT.md     # context เท่าที่ production เห็น

ทั้งหมด gitignored (ข้อมูลลูกค้า) — สิ่งที่ commit คือโค้ด framework, spec การตรวจ และตัวเลขรวมใน SCOREBOARD.md

  segment_id: seg-004
  files: [input/ค่าเช่า.pdf]
  pages: "1-2"

provenance:
  verified_by: answer_key(034) + ledger_gate
    + direct document inspection 2026-07-11

กฎเหล็กของ expectation เฉลย = "สิ่งที่ agent ที่ถูกต้องควรตอบตาม contract ของมัน" ไม่ใช่ "ความจริงในอุดมคติ" — เอกสาร 033 ที่ไม่พิมพ์ WHT: เฉลยของ watson คือ wht: null บวกธง แม้ answer key จะ book WHT 691.88 เพราะการแปลงธงเป็นตัวเลขคือหน้าที่ของชั้น policy ไม่ใช่ leaf agent — เกรด leaf ด้วยผลปลายน้ำเท่ากับลงโทษพฤติกรรมที่ถูก

ลำดับความน่าเชื่อของเฉลยตัวอย่างจริง
1 · ตัวเลขที่ตรงกันระหว่าง answer key กับ run ที่ผ่าน reviewยอด/VAT/เลขที่ของใบรายได้ RT-202605xxxxx ทั้ง 4 ใบ
2 · การเปิดเอกสารจริงดูด้วยตาซูม 600dpi ยืนยัน DATE = 05.05.2026 และเลขที่ IVT-20260500096
3 · ค่าจาก run ที่ human review ผ่าน (ฟิลด์ที่เฉลยไม่ cover)page_disposition, review flags

harvest.ts ปฏิเสธทุกเดือนที่ ledger gate ไม่ pass เคสที่เฉลยยืนยันไม่ครบติดป้าย provisional — ยังรันได้แต่ไม่ถูกนับใน silent-error rate ตัวหลัก เมื่อพบว่าเฉลยผิดเอง: แก้ + bump VERSION + รัน baseline ใหม่ — ห้ามแก้เฉลยเงียบๆ

รายการ 06เกณฑ์การให้คะแนน — สี่สถานะต่อฟิลด์

ตราประทับความหมายระดับอันตราย
correct ตรงเฉลยใน tolerance — เงิน ±0.01, วันที่/เลขภาษีตรงตัว, ชื่อบริษัทเทียบแบบ normalize, wht ถือ null≡0 —
wrong_flagged ผิด แต่มี review flag ที่พูดถึงฟิลด์นั้น ต่ำ — คนตรวจเห็นธงแล้วตามไปดู
wrong_silent ผิดโดยไม่มีธง สูงสุด — เดินเข้า PEAK โดยไม่มีใครเอะใจ สามแบบนี้รวมกันคือ silent-error rate
missing / spurious ค่าที่ควรมีแต่หาย / ค่าที่เอกสารไม่มีแต่แต่งขึ้น

ฟิลด์ critical ของ watson: document_no · document_date · seller/buyer_tax_id · gross_total · vat · wht · direction · doc_kind · page_disposition — ตัวสุดท้ายคือคำตอบของ "ตัดสิน skip หน้าไหนถูกไหม" ส่วน sherlock วัด cluster_membership · bookable_docs · multi_member_clusters ด้วยการจับคู่ member-set แบบ multiset

รายการ 07วัดความนิ่ง — ไม่ใช่แค่ความถูก

--replicates 3 จะ spawn subagent 3 ตัวขนานกันด้วย prompt เดียวกัน (context แยกขาด ไม่เห็นคำตอบกัน — ผลทางสถิติเท่ารันตามลำดับ แต่ wall-clock เท่ารอบเดียว) แล้ว grader เพิ่มมิติ agreement รายฟิลด์:

Agreementอ่านว่าทางไป
3/3 ตรงกันฟิลด์นิ่งเชื่อผลรอบเดียวได้ในอนาคต
2/3ฟิลด์สั่นคลอน — ปัญหาคือความไม่นิ่งของการอ่าน ไม่ใช่ rule ผิดวิธีแก้คนละทางกับแก้ prompt (เช่น ต้องซูมภาพ)
ต่างกันหมดสุ่มจริงจังอย่าเพิ่งแก้ rule — หา root cause ของความสุ่มก่อน

ค่า token จ่าย ×N ตรงๆ — ใช้เป็นโหมดเสริมตอนสงสัยความนิ่งหรือตอนเคาะ baseline ใหม่ รอบพัฒนา prompt ปกติรันเดี่ยวพอ

รายการ 08กรณีศึกษา: ไล่ล่า "วันที่ผี" บนใบเช่าสแกน

ใบเช่า Golden Land เป็นสแกนความละเอียดต่ำ พิมพ์ DATE 05.05.2026 แต่ watson อ่านเป็น 2026-05-03 เป็นบางรอบ — โดยมั่นใจเต็มร้อย ไม่ยกธง นี่คือ silent error ตัวอย่างตำรา และ eval ทำให้การไล่ล่าเป็นวิทยาศาสตร์:

รอบสมมติฐาน / rule ที่ใส่ผลวัด (3 replicates)บทเรียน
0ยังไม่มี rule 2/3 จับ failure mode ได้: บางรอบหยิบวันที่ service period บางรอบอ่านเลขผิด
1playbook: document_date = วันที่ออกใน header เท่านั้น ห้ามใช้ช่วงเวลาบริการใน line item รันเต็มผ่าน แต่ stability ยัง 2/3 ปิดได้หนึ่งประเภท (service period) แต่หลักเลข 05↔03 ยังสั่น
2"ซูมเมื่อรู้สึกไม่แน่ใจ" 1/3 — แย่ลง! rule มีเงื่อนไขไม่ทำงาน เพราะการอ่านผิดมันรู้สึกมั่นใจ — agent ไม่เคยรู้สึกว่าต้องซูม
3บังคับซูม: ทุกหน้าสแกน (เช็ค text layer ด้วย pdftotext | wc -c) ต้อง render header 300dpi แล้วอ่านเลขที่+วันที่จากภาพซูมก่อนจด 3/3 ทุกฟิลด์ agree ปิดจ๊อบ — replicate หนึ่งรายงานเองว่า "ซูมแล้วยืนยัน 05.05.2026 ไม่ใช่ 03 ที่ความละเอียดต่ำหลอก"

ถ้าไม่มี eval รอบ 2 ("ซูมเมื่อไม่แน่ใจ") ฟังดูสมเหตุสมผลมาก และคงถูก commit ไปทั้งที่มันแย่กว่าไม่มี rule — ตัวเลข 1/3 เท่านั้นที่หยุดเราไว้

รายการ 09สิ่งที่ระบบตรวจจับได้เอง — นอกเหนือจากบั๊ก agent

สิ่งที่เจอใครผิดผลลัพธ์
เลขที่เอกสาร IVT-20260500096 ไม่ตรงเฉลย IVT-20260600057 answer key คีย์ผิดเอง (เอกสารจริงพิมพ์ …500096 ชัด) watson พ้นข้อกล่าวหา — เคสอัปเกรดจาก provisional เป็น solid
เคส b3 ขาดไฟล์ duplicate 1 ไฟล์ คนออกข้อสอบ (harvest จาก documents[] แทน page_disposition) แก้เคส + เพิ่ม guard ใน harvest.ts ให้ refuse เคสที่ input ไม่ครอบ disposition
review_flags บางรอบเป็น object บางรอบเป็น string schema ของ skill ไม่ pin รูปแบบธง grader ทนทั้งสองแบบ + จดเป็น candidate ให้ tighten validator
draft พิษ: merge ใบสำคัญจ่ายคนละเรื่องด้วยเลขเล่ม + อ้างเอกสารที่ไม่มีจริง โจทย์หลอกโดยบังเอิญ sherlock grep ตรวจก่อน refuse — ได้ pattern การออกโจทย์ adversarial สำหรับรอบหน้า
เคสใหญ่ 403 ธุรกรรม: 7 สถานการณ์ไม่ตรงเฉลย กระจุกที่เอกสาร doc_no ซ้ำ/null เฉลยเองยัง verify แค่ระดับ booking ไม่ใช่ระดับ cluster downgrade เคสเป็น provisional — สถานการณ์ที่ adjudicate แล้วจะกลายเป็น mini case ใหม่

ข้อสังเกต ครึ่งหนึ่งของสิ่งที่จับได้คือความผิดของ "เฉลย" และ "ข้อสอบ" ไม่ใช่ตัว agent — ระบบวัดที่ดีตรวจสอบตัวมันเองไปด้วย (self-test: เอาเฉลยเป็นคำตอบต้องได้ 100% · negative test: ปลอมค่าผิดต้องจับได้ครบทุกประเภท)

รายการ 11ปุ่มปรับจูน — อาการแบบไหน แก้ที่ไฟล์ไหน

ระบบนี้มีของให้ปรับอยู่ไม่กี่จุด และแต่ละจุดมีกติกากำกับของมัน — ตารางนี้คือแผนที่ "เห็นอาการนี้ → หมุนปุ่มไหน" ทั้งหมดที่มี:

อาการที่เห็นจากรายงานปุ่มที่หมุนกติกากำกับ
wrong_silent บนเคส solid — agent อ่าน/ตัดสินผิดจริง playbook ของ agent: .claude/agents/ksk-watson.md ฯลฯ — เพิ่ม/แก้ rule (เช่นกฎบังคับซูม) วัดก่อน–หลังเสมอ และเขียนเป็นกฎทั่วไป ไม่ใช่เรื่องเล่ารายเคส; rule มีเงื่อนไข ("เมื่อไม่แน่ใจ") มักไม่ทำงาน — ดูรายการ 08
ผิดเพราะ "สำนวน" ไม่ใช่ booking — null vs 0, ชื่อสะกดต่าง, doc_kind คนละคำแต่ book เหมือนกัน เกณฑ์ตรวจใน evals/specs/watson.ts: tolerance ±0.01, NULL_ZERO_EQUIVALENT, DOC_KIND_ALIASES, FLAG_KEYWORDS แก้ spec แล้วต้องรัน self-test (เฉลย→100%) และ negative test (ค่าปลอม→จับได้) ซ้ำก่อนเชื่อผล
ธงถูกยกจริงแต่เกรดเป็น silent — keyword ไม่ match ภาษาที่ agent ใช้ FLAG_KEYWORDS ใน spec (จับ substring ไทย/อังกฤษต่อฟิลด์) เพิ่ม keyword ให้กว้างขึ้นได้ แต่ระวังกว้างจน flag อะไรก็ได้ถือว่าพูดถึงทุกฟิลด์
ไล่ดูแล้ว agent ถูก — เฉลยผิดเอง (เคยเกิดจริง: answer key คีย์เลขเอกสารผิด) expected.json / case.yaml ของเคสนั้น แก้ได้ แต่ต้อง bump VERSION + รันใหม่ + pin baseline ใหม่ — ห้ามแก้เฉลยเงียบๆ
failure mode ใหม่ที่ยังไม่มีข้อสอบครอบ เพิ่มเคส: harvest.ts จากเดือนที่ ledger gate ผ่าน หรือออกแบบ mini-case must/must-not เอง (แบบ sherlock) เคสใหม่เริ่มที่ provisional: true จน verify ครบ — ไม่ปนเข้า rate หลัก
ผลสั่นข้ามรอบ — รอบนี้ผ่าน รอบหน้าตก ทั้งที่ไม่ได้แก้อะไร --replicates 3 แล้วอ่านตาราง agreement รายฟิลด์ อย่าแก้ rule จากผลรันเดียว และห้าม cherry-pick replicate ที่สวยที่สุด
อยากแก้ prompt ที่ eval ส่งให้ agent แก้ template ฝั่ง production (SKILL.md ของ workflow) แล้วให้ dispatch.ts mirror ตาม ห้ามแก้ฝั่ง eval ฝ่ายเดียว — prompt ที่เพี้ยนจาก production ทำให้ตัวเลขทั้งกระดานไร้ความหมาย
run ใหม่ดีจริงและอยากให้เป็นเส้นเทียบถัดไป bun run report.ts -- <agent> --run <id> --set-baseline + เพิ่มแถวใน SCOREBOARD.md ตั้งเมื่อคนยืนยันแล้วเท่านั้น — baseline คือคำสัญญาว่า "ต่ำกว่านี้คือ regression"

หลักคิดเวลาเลือกปุ่ม ไล่จากถูกไปแพง: สงสัยเกณฑ์ตรวจก่อน (spec) → แล้วสงสัยเฉลย/ข้อสอบ (case) → ค่อยสงสัยตัว agent (playbook) — ประสบการณ์จริงในรายการ 09 บอกว่าครึ่งหนึ่งของ "agent ผิด" ที่เห็นครั้งแรก จบที่เฉลยหรือข้อสอบผิดเอง

Goal / Non-goals

ประเภทรายละเอียด
Goalวัด agent เดี่ยวบน fixture verified ด้วย grader deterministic; ตัวเลขหลัก = silent-error rate (เคส solid); กัน regression ด้วย baseline diff ที่ exit 1
Goaldispatch prompt ต้อง mirror template ใน SKILL.md แบบ verbatim ผ่าน harness เดียวกับ production (Agent tool)
Non-goalไม่ใช้ LLM-as-judge, ไม่ใช้ promptfoo, ไม่ commit ข้อมูลลูกค้า (dataset/ผลดิบอยู่ใต้ samples/evals/ ซึ่ง gitignored)
Non-goalไม่แทน Ledger Gates หรือ human review ของ production run — eval เป็นเครื่องมือพัฒนา skill

การเพิ่ม agent ใหม่ (เช่น poirot) — หกขั้น

ขั้นงานเกณฑ์เสร็จ
1เขียน specs/<agent>.ts — นิยามฟิลด์/หน่วยที่เกรด, tolerance, การจับคู่type-checks; หน่วยการเกรด map เข้า field_states + counts มาตรฐานได้
2เพิ่ม branch ใน harvest/dispatch/grade (ดูตัวอย่าง sherlock ซึ่ง output path ตายตัว → ต้อง clone ต่อ replicate)dispatch พิมพ์ prompt ตรง template production ของ stage นั้น
3Self-test: เอา expected เป็น output100% pass, 0 silent
4Negative test: ปลอมค่าผิดอย่างน้อย 3 ประเภท (wrong/missing/spurious)จับได้ครบ ถูกประเภท
5Harvest เคสจริง (gate pass เท่านั้น) — เคสเล็กที่ออกแบบ must/must-not ไว้ล่วงหน้าเคสแรก solid ≥ 1, ที่เหลือ provisional ติดป้ายเหตุผล
6รัน live → ตรวจ failure ทีละตัว (อาจเป็นเฉลย/ข้อสอบผิด) → pin baseline → ลง SCOREBOARDbaseline.json + แถวใหม่ใน SCOREBOARD.md

Dataset hygiene — กฎที่ห้ามแหก

  • harvest เฉพาะเดือนที่ ledger gate = final/pass และเขียนแหล่ง verify ใน provenance เสมอ
  • เคส watson: ไฟล์ input ต้องครอบทุกหน้าใน expected.page_disposition (harvest guard บังคับแล้ว)
  • เฉลยเปลี่ยน = bump VERSION + รัน baseline ใหม่ — ห้ามแก้เงียบ
  • production run ห้ามอ่าน samples/evals/ และ eval case เป็นสำเนา self-contained ไม่ชี้กลับ client folder
  • silent-error rate ตัวหลักนับเฉพาะเคส solid; provisional รายงานแยก
  • ผลดิบทุก run เก็บถาวรใต้ _runs/ ไม่ทับกัน; commit เฉพาะตัวเลขรวม
